# Connectors

Specifications for pulling from source systems (HubSpot, Salesforce, Gong, Fathom, Confluence, Notion, Stripe, Slack, email) and generating wiki updates via PR.

**v1 ships no real connector implementations.** This directory holds the **abstract pattern** in [`PATTERN.md`](PATTERN.md). Real implementations are built per-fork against the operator's specific source systems, or forked from community contributions.

---

## Why abstract-only in v1

- Scope discipline. The framework is the deliverable, not integrations.
- Each connector is a meaningful project (auth, rate limits, pagination, error handling, PII, schema mapping). Building all of them well would be a separate body of work.
- Real connectors are meaningful against a specific operator's source systems and data shape. Best built per-fork or as community contributions.
- The community contribution model benefits from a clear pattern to conform to.

## The connector contract

Every connector implementation must:

1. **Authenticate** via credentials pulled from environment (never committed).
2. **Extract** data from the source system in a well-defined query.
3. **Synthesize** the extracted data into markdown (summarize, structure, redact PII).
4. **Snapshot** the extraction to `sources/<category>/YYYY-MM-DD-<slug>.md` (if in-repo per [`../../sources/POLICY.md`](../../sources/POLICY.md)) or record the citation as a system-query reference.
5. **PR.** Open a PR against the wiki with proposed changes. The PR description cites the snapshot plus the connector run ID.

Full pseudo-code: [`PATTERN.md`](PATTERN.md).

## Citation convention

Two citation formats map to the two source models:

### In-repo file reference (Model A)

```markdown
[source: sources/calls/2026-04-15-acme-discovery.md]
```

Used when the connector synthesized in-repo content.

### System-query reference (Model B)

```markdown
[source: hubspot://deals?stage=closed-won&period=2026-Q1 as_of:2026-04-19]
```

Used when the source stays in the system of record. The `<system>://` protocol prefix must match a connector spec in this directory. `ci verify` enforces this.

## Webhook vs. pull

Two ingestion patterns. Connector implementations document which they use.

- **Webhook.** The source system pushes events to an endpoint and the connector processes them in near-real-time. Lower latency. Requires infrastructure (endpoint, queue). Preferred for high-signal events (deal stage change, call transcript ready).
- **Pull.** Scheduled poll. The connector queries the source on a cadence. Simpler to implement. Higher latency. Preferred for bulk data (all calls from last week, all deals in this segment).

Most connectors combine both: webhook for immediate triggers, pull for backfill and batch analysis.

## Lifecycle of a connector run

```
[trigger: webhook event OR scheduled poll]
        │
        ├── auth against source system
        ├── extract matching data
        ├── synthesize into markdown + structured metadata
        ├── redact PII per sources/POLICY.md
        │
        ▼
[decision: in-repo snapshot vs. system-query citation]
        │
   ┌────┴────┐
   │         │
 [in-repo] [system-query]
   │         │
   ├── write sources/<cat>/<date>-<slug>.md
   ├── set frontmatter.source_captured_at = now
   │
   └─────────┬─────────┘
             │
             ▼
[compute wiki impact]
        ├── which existing pages are affected?
        ├── which new page(s) to create?
        ├── any contradictions with current state?
        │
        ▼
[open PR]
        ├── diff = page updates + new source file (if in-repo)
        ├── description cites the connector run, source(s), affected pages,
        │   confidence score, contradiction flags
        ├── routes to reviewer per ingestion/pr-workflow.md
        │
        ▼
[reviewer merges or rejects]
        ├── log.md gets append via log-append.yml
        └── drift detection picks up any new source_drift flags
```

## What connectors never do

- **Write to `log.md` directly.** That is automation-appended on merge.
- **Write to `index.md` directly.** That is regenerated by `ci reindex`.
- **Bypass PR.** Every connector change is a PR, regardless of maturity stage.
- **Store credentials in the repo.** Always environment or secret manager.
- **Duplicate live-system data into the repo.** Follow the hybrid policy in [`../../sources/POLICY.md`](../../sources/POLICY.md).

## Implementation plan for real connectors

Candidate first connectors:

| System | Priority | Pattern |
|---|---|---|
| HubSpot | High | Pull for deal snapshots; webhook for stage change |
| Salesforce | High | Same as HubSpot |
| Gong / Fathom | High | Webhook on transcript ready; pull for backfill |
| Confluence | Medium | Webhook on page change; pull for initial sync |
| Notion | Medium | Same as Confluence |
| Stripe | Medium | Pull for usage aggregates; webhook for material billing events |
| Slack | Lower | Pull with operator-scoped channels only |
| Email / Gmail | Lower | Opt-in per mailbox; heavy PII filtering |

Each ships as its own PR with a `<system>.md` connector spec in this directory, an implementation reference (in a separate repo or a `connectors/impl/` subdir if it belongs here), and runbook.

## Community contributions

The connector pattern is designed for community extension. If you build a connector for a system we haven't covered:

1. Fork the repo, implement your connector following `PATTERN.md`.
2. Open a PR adding a `<system>.md` spec to this directory.
3. Link the implementation (even if it lives in a separate repo).
4. Include a smoke test.
5. Tag `@sikkrasmus` (or current maintainer) for review.
