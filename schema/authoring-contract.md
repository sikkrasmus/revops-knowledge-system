# Authoring Contract

The canonical operating agreement between human authors, LLM authors, and reviewers. Every page in this corpus — shared, function-specific, or source snapshot — is authored under this contract. `ci verify` enforces the mechanical parts; the judgmental parts are enforced by PR review.

This document is hand-written. Updates require a PR with changelog.

---

## 1. Who writes what

- **Humans** set the schema (this file, `frontmatter.md`, `section-labels.md`, `enums.yaml`, page templates), curate sources, and review PRs.
- **LLMs** draft page content, open PRs with diffs, and cite sources. LLMs never edit `index.md` or `log.md` directly (see §6).
- **Automation** regenerates `index.md` on every push (`ci reindex` in CI) and appends to `log.md` on PR merge and lint runs.

When a human edits a page directly, they follow the same contract — frontmatter, citations, section labels, PR etiquette.

## 2. Source-precedence rules

When two sources disagree, apply precedence in order:

1. **Signed, dated, authoritative** (contracts, legal docs, board-approved pricing) beats everything below.
2. **Recorded conversations with named parties** (Gong/Fathom calls, meeting recordings) beat everything below.
3. **Written internal artifacts** (Confluence, Notion, Drive docs) if recent (<90 days) and authored by a stakeholder with authority on the topic.
4. **Synthesized summaries** (already-condensed notes, wiki pages from earlier synthesis).
5. **Email / Slack / chat** — only when corroborated by a higher tier, except for explicit authoritative approvals.

Recency breaks ties within a tier: newer wins if nothing else distinguishes.

**When precedence cannot resolve**: open a PR with both sources cited, flag `has_contradiction: [<other-page>]`, and hand the call to the page's reviewer.

## 3. Frontmatter requirements

Every page MUST have YAML frontmatter. Spec: [`frontmatter.md`](frontmatter.md). Enum values: [`enums.yaml`](enums.yaml).

**Required on every page**: `type`, `title`, `description`, `owner`, `last_reviewed`.

**Additionally required on `shared/` and `functions/` pages**: `verified_until`, `reviewers`.

**Required where applicable**: `motion`, `segment`, `consumers`, `sources`.

`description` is a single-line summary used by `ci reindex` to build `index.md`. Max 200 chars, ends with `.`. Write it carefully — it's what agents read first when retrieving.

## 4. Citation requirements

Every factual claim carries a claim-level citation inline:

```markdown
Multi-product platforms close 40% faster than single-product peers
[source: sources/calls/2026-04-15-acme-discovery.md].
```

Every cited source also appears in the page's frontmatter `sources:` list. `ci verify` fails on any `[source: ...]` whose reference isn't in `sources:`, and on any `sources:` entry not cited somewhere in the body.

Two source formats:

- **File reference**: `sources/<category>/<filename>` — must resolve to a file in the repo.
- **System query**: `<system>://<query-string> as_of:<ISO-date>` — `<system>` must have a connector spec in `ingestion/connectors/`.

No unsourced claims survive review. If you can't cite a source, either find one or mark the claim `confidence: low` and open a follow-up issue.

## 5. Conflict-resolution policy

When a new source contradicts an existing page:

1. Do not overwrite silently. Mark the page with `has_contradiction: [<source-path-or-page>]` in frontmatter.
2. Open a reconciliation PR with both sources cited, the proposed resolution, and a confidence score.
3. Route to the page's reviewer (see `ingestion/pr-workflow.md`).
4. Merge only after human decision. Append to `log.md` via the merge hook.

The LLM never unilaterally decides which source wins on strategic content (ICP, pricing, positioning, legal, packaging). Those always PR, always to a human.

## 6. Never-edit-directly files

Two files are automation-maintained:

- **`index.md`** — regenerated by `ci reindex` from page frontmatter. LLMs never edit it. Humans never edit it. Instead: write a good `description` in each page's frontmatter; `ci reindex` does the rest. CI fails the build on a stale `index.md`.
- **`log.md`** — appended by `.github/workflows/log-append.yml` on PR merge, by the lint pass on completion, by the ingest pipeline on completion. Human edits to `log.md` are not permitted except at bootstrap (the initial entry in this repo's `log.md`).

## 7. Staleness handling

Three freshness signals are first-class:

- **`verified_until: <date>`** — when today > `verified_until`, `ci verify` sets `verification_stale: true` and surfaces the page in `ci stale`.
- **`source_drift: true`** — set by the drift-detection lint when a source has changed since snapshot. Spec: [`../ingestion/drift-detection.md`](../ingestion/drift-detection.md).
- **`has_contradiction: <list-of-page-paths>`** — set by the lint when another page cites a conflicting claim.

When a signal fires, the LLM opens a reconciliation PR. It does not auto-rewrite the page.

## 8. PR requirements

Every LLM-authored PR includes:

- **Diff** with explicit file paths.
- **Sources cited** at the PR description level (summary of which sources informed the change).
- **Affected consumers** — which functions / pages cite the changed page. Use `ci graph` to enumerate.
- **Confidence score** — `low` / `medium` / `high` (heuristic in v1).
- **Reviewer routing** — per `ingestion/pr-workflow.md`, based on directory and change class.
- **Lint status** — attach `ci verify` output.

When to PR vs. auto-merge: see `ingestion/auto-merge-rules.yaml` (concept only in v1 — Stage 1 means 100% PR).

## 9. Page creation, rename, delete

- **New page** — always PR, never auto-merge, regardless of maturity stage. Include rationale in the PR description: why this page now, which function(s) consume it, which sources back it.
- **Rename** — always PR. Requires updating every inbound link (`ci graph` enumerates them). Breaking a slug breaks retrieval.
- **Delete** — always PR with issue linked. Provide the rationale, the replacement page (if any), and the inbound-link cleanup. Never delete without an issue.

## 10. Voice and style

- **Direct, declarative, unhedged** when citing a claim that's well-sourced. Hedge explicitly (`~`, `approximately`, `estimated`) when the source itself hedges.
- **No marketing voice on internal pages.** Internal pages are reference material for practitioners, not pitches.
- **One claim per sentence where it's load-bearing.** Makes lint passes and drift detection tractable.
- **No emojis on internal pages** unless the section is explicitly reproducing a customer-facing artifact (e.g., an elevator pitch quoted in the positioning page).
- **Present tense for current state**, past tense for historical events (closed deals, deprecated pricing).

## 11. Things to never do

- Never overwrite a page silently. Always diff-and-PR.
- Never edit `index.md` or `log.md` directly.
- Never cite a source not in the `sources:` frontmatter.
- Never delete a page without an issue.
- Never merge a PR that touches ICP, pricing, packaging, legal, or strategic positioning without a human reviewer's approval.
- Never suppress a `has_contradiction` flag by editing the conflicting source out.
- Never reduce `verified_until` without a re-verification note.
- Never add a new enum value (type, motion, segment, consumer, confidence) without a PR to `schema/enums.yaml`.

## 12. When in doubt

Flag, don't guess. Add a `confidence: low` frontmatter marker, open a reconciliation issue, and let a human resolve. The cost of surfacing uncertainty is low; the cost of confidently-wrong content compounds across every downstream consumer.
